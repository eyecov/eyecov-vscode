# Coverage Model

This document defines the **canonical EyeCov coverage model** and the data flow
that turns external coverage artifacts into editor features and MCP responses.

For the technical structure of the extension, see
[COVERAGE_ARCHITECTURE.md](COVERAGE_ARCHITECTURE.md).
For user-visible behavior, see [COVERAGE_FEATURES.md](COVERAGE_FEATURES.md).
For MCP tool contracts, see [MCP_SERVER.md](MCP_SERVER.md).

## Why this document exists

EyeCov is not just “a VS Code extension that reads coverage”.

It has a small internal platform shape:

- **coverage artifacts** from external tools
- **adapters** that parse and normalize them
- a shared **runtime coverage model**
- **consumers** in the editor and MCP server

That model is the center of the system. This document is the single place that
defines it.

## Canonical flow

The high-level flow is:

1. A coverage tool writes an **artifact**.
2. An EyeCov **adapter** finds coverage for a source file.
3. The adapter normalizes that result into a shared **coverage record**.
4. Aggregation and cache layers build broader **path/project views** when needed.
5. The editor and MCP server consume the same normalized data.

In shorthand:

```text
coverage artifact
    ↓ parse
adapter
    ↓ normalize
coverage record
    ↓ aggregate / cache
editor + MCP consumers
```

## Coverage artifact

A coverage artifact is any external output generated by a test or coverage tool.

Current examples in EyeCov:

- PHPUnit HTML
- LCOV

Potential future examples:

- Cobertura XML
- Clover XML
- PHPUnit XML variants
- language-specific JSON formats

Artifacts differ wildly in structure. EyeCov should never let that leak into
editor logic or MCP tool logic.

## Adapter

An adapter is the format-specific bridge between an external artifact and the
EyeCov runtime model.

Current adapters include:

- `PhpUnitHtmlAdapter`
- `LcovAdapter`

Adapter responsibilities:

- locate the right artifact for a source file
- parse coverage data for that file
- normalize paths
- reject stale artifacts
- return a normalized coverage record

Adapters should stay format-specific. They should not contain editor decoration
logic, MCP response formatting, or path/project aggregation logic.

## Coverage record

The normalized per-file unit in EyeCov is the coverage record.

Current implementation (see `src/coverage-resolver.ts` and related types) is
editor-oriented and optimized for fast lookup:

- `sourcePath`
- `coveredLines`
- `uncoveredLines`
- `uncoverableLines`
- `lineCoveragePercent`
- optional `coverageHtmlPath`
- optional `testsByLine`
- optional `lineStatuses`

Conceptually, the record answers:

- Which file is this?
- Which executable lines are covered?
- Which executable lines are uncovered?
- Which lines are uncoverable / warning / ignored?
- What summary percentage is available?
- Does the format provide covering-test data?

## Freshness model

Coverage is only useful when it still matches the source code.

EyeCov therefore treats freshness as part of the model:

- if the source file is newer than the artifact, coverage is **stale**
- stale coverage is rejected before it reaches the editor or MCP tools
- if paths cannot be read safely, EyeCov fails closed and returns no coverage

This keeps the shared model trustworthy.

## Aggregation model

Per-file coverage records can be aggregated into broader views.

Current aggregate consumers:

- `coverage_path`
- `coverage_project`
- `coverage_test_priority`

Aggregation answers questions like:

- What is coverage under this folder or path prefix?
- Which files are the worst covered?
- Which files have zero coverage?
- Where should tests be added first?

Aggregation should build on normalized records rather than re-implementing
format-specific parsing rules.

## Cache model

EyeCov optionally materializes normalized results into a cache:

- per-workspace cache file under `.eyecov/coverage-cache.json`
- per-file normalized entries
- pre-aggregated project totals

The cache exists to speed up broader path/project queries and MCP usage. It is
an optimization layer, not a separate source of truth.

## Consumers

### Editor

The editor uses coverage records to render:

- line highlighting
- gutter markers
- status bar summaries
- file-level coverage info

### MCP server

The MCP server uses the same runtime and records to expose:

- file coverage
- covering tests by line
- path/project aggregates
- test priority suggestions

### Future consumers

The same model could later support:

- CLI tools
- CI reporting helpers
- diff coverage workflows
- project analytics

## Boundaries

A healthy EyeCov boundary line looks like this:

- **format-specific concerns** stay in adapters and parser docs
- **shared coverage semantics** live in the coverage model
- **editor rendering** stays in extension/editor code
- **tool contracts** stay in MCP docs and MCP response shaping

That separation keeps the codebase understandable for contributors and much
easier for AI tools to navigate.
